# ⛔ СТРОГО. Нарушение = отказ от правки.

Правило: `.cursor/rules/zero-initiative-strict.mdc` — **не обходить.**

**На всегда:** любое сообщение → **короткий конспект** → «Правильно понял? Делать?» → ждать **«да»** → только потом код. Даже «сделай» / «почини» в том же сообщении — **сначала конспект, не код.**

1. Без **«да» / «делай»** — **не менять файлы**.
2. **Только** последнее сообщение. Ничего «заодно», «логично», «связанно».
3. **Конспект (короткий):** понял → сделаю → не трогаю → **«Правильно понял? Делать?»**
4. **Не менять** файлы и поведение, **не названные в запросе**.
5. **Рабочий код** — только с **«можно тронуть X»**.
6. **Не понял** — спросить. **Не кодить.**
7. **Одна задача = один минимальный diff.**
8. **Никакой самодеятельности** — никаких решений, правок и добавок от себя.

# Формат работы — одобрение до правок (строго, без обхода)

**Короткий конспект → «Правильно понял? Делать?» → «да» → правки.** Даже прямой приказ в сообщении — сначала конспект, не код.

После **каждой** заявки на исправление или редактирование:

1. **Изучить** тему (код, контекст). **Не править** файлы.
2. **Короткий конспект:**
   - понял;
   - сделаю (файлы);
   - не трогаю;
   - **«Правильно понял? Делать?»**
3. **Ждать одобрения** («да» / «делай») — пользователь **прочитал** конспект.
4. **Только после одобрения** — правки.

До одобрения: никаких изменений в коде, конфигах и «улучшений заодно».

# Работа по запросу пользователя

**Только то, что просят.** Делай ровно одну запрошенную вещь. Не меняй «заодно» тайминги, скорости, соседний UI, рефакторинг, «улучшения» или другие части сцены.

**Связанные правки — с отчётом.** Если без правки другого места задачу не выполнить, в ответе явно напиши:
- какой файл или участок пришлось тронуть;
- почему это связано с запросом;
- что именно там изменилось.

**Перед правкой** кратко перечисли, что меняешь. **После** — не добавляй ничего сверх запроса, если пользователь сам не попросил.

**Жёстко:** нужно изменить **уже работающий** код ради задачи → **спроси разрешение**, жди «да». **Доделываешь** текущую задачу, не трогая остальное работающее → можно без отдельного спроса.

# Не ломать то, что уже работает

- Не удаляй и не отключай рабочий UI, сценарии, кнопки и эффекты без явной просьбы пользователя.
- **Жёстко:** если для задачи нужно изменить уже работающий код — **сначала спроси разрешение**, потом правь. Без «да» не трогай.
- **Исключение:** доделываешь текущую запрошенную задачу и не лезешь в другие работающие части — можно продолжать без отдельного спроса.
- При рефакторинге или добавлении нового поведения сохраняй существующий рабочий поток; если что-то нужно убрать или заменить — сначала согласуй.
- Если правка затрагивает соседний код, проверь, что прежнее поведение не регресснуло.

# Точки сферы — всегда 8000 (жёстко)

- `POINT_COUNT = 8000` в `components/SpherePoints.tsx` — **не менять**.
- **После взрыва и дальше — те же 8000.** Не добавлять, не убирать, не заменять. Всегда оперируем этой массой целиком.
- Все фазы (`sphere`, `explode`, `orbit`, `reside`, любые новые) двигают **все 8000** точек каждый кадр.
- Без подмножеств, LOD, spawn/cull и вторых наборов частиц сферы.
- Комета (`COMET_POINT_COUNT`) — отдельный объект, не точки сферы.

# Один длинный лендинг (жёстко)

- Сайт — **одна логическая страница**: все функции (комета, курсор, магнит, сценарий, 8000 точек) работают **одинаково на любой секции**.
- Визуально лента разбита на секции (`100vw`) — скролл двигает только **контент** (`site-track`: текст, кнопки, блоки). Snap к секции с большей видимостью.
