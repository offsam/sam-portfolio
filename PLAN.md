# Конспект эффектов для сайта-портфолио (рекламы самого себя)

Источник — личный разбор референсов. Ниже — что именно понравилось и как это технически называется/реализуется.

---

## 1. benjaminrobinet.com
**Что увидел:** Образы "собираются" из частиц. Чёрно-белая картинка — при клике становится цветной/прозрачной и появляется описание проекта.
**Техническое название:** Particle morphing / image dissolve-in, hover/click reveal с переходом ЧБ → цвет + текстовый оверлей.
**Стек:** WebGL (Three.js/OGL) для частиц, CSS transition или canvas crossfade для ЧБ→цвет.

## 2. bruno-simon.com
**Что увидел:** Управление машинкой через кнопки как в игре — сайт превращается в мини-игру, разделы открываются через геймплей.
**Техническое название:** 3D scene navigation / portfolio-as-a-game (физика на Cannon.js/Rapier + Three.js).
**Стек:** Three.js + физический движок. Это самый трудозатратный вариант из всех.

## 3. thomasaufresne.com
**Что увидел:** Частицы следуют за курсором (как на mvp-code.ru).
**Техническое название:** Particle cursor trail.
**Стек:** уже разобрали ранее — threejs-toys `particlesCursor` или кастомный GPGPU-шейдер.

## 4. aristidebenoist.com
**Что увидел:** Скролл идёт не вниз, а вбок (горизонтальный скролл).
**Техническое название:** Horizontal scroll hijacking.
**Стек:** GSAP ScrollTrigger (`scrollTrigger: { horizontal: true }`) или Locomotive Scroll/Lenis + transform translateX по скроллу.

## 5. patrickheng.com
**Что увидел:** Тоже боковой скролл + постоянная фоновая анимация, которая не останавливается.
**Техническое название:** Horizontal scroll + ambient background animation (фон живёт независимо от скролла/курсора).
**Стек:** то же + отдельный requestAnimationFrame-луп для фона (canvas/WebGL), не привязанный к скролл-событиям.

## 6. clarissemichard.com
**Что увидел:** Курсор влияет на весь текст на экране (искажение/реакция), плюс сам курсор визуально необычный.
**Техническое название:** Text distortion on cursor proximity + custom cursor shape.
**Стек:** SplitText (GSAP) на буквы/слова + расчёт расстояния до курсора → transform/blur по каждой букве; кастомный курсор на Framer Motion/CSS.

## 7. thomasmonavon.com
**Что увидел:** Два эффекта:
(а) Скролл-снэппинг — если остановился между блоками, авто-доезжает до того блока, который занимает больше экрана.
(б) Переход между картинками — одна "закрывается" фоном на месте, с другой стороны "открывается" другая (не обычный fade, а wipe/reveal).
**Техническое название:** (а) Scroll snap с порогом (CSS `scroll-snap` или GSAP ScrollTrigger snap); (б) clip-path/mask wipe transition между секциями.
**Стек:** CSS scroll-snap-type + JS-порог, либо GSAP ScrollTrigger `snap`; для перехода — `clip-path` анимация или WebGL shader-transition.

## 8. evanfasquelle.com
**Что увидел:** Двойное движение при скролле — основной контент идёт вниз, а бегущая строка на фоне одновременно ускоряется и едет в свою сторону (слева), создавая эффект "два слоя живут в разном темпе".
**Техническое название:** Parallax с разной скоростью слоёв + marquee, скорость которого зависит от скорости скролла.
**Стек:** CSS marquee/бегущая строка + JS, где скорость анимации привязана к `scroll velocity` (можно через Lenis `onScroll` дельту).

## 9. kentatoshikura.com
**Что увидел:** Скролл идёт вниз, но картинки внутри круга/маски одновременно двигаются вправо — небольшой оптический "обман".
**Техническое название:** Masked element with independent horizontal motion during vertical scroll.
**Стек:** контейнер с `overflow: hidden`/круглой маской, внутреннее изображение анимируется по X через GSAP, привязанное к прогрессу вертикального скролла.

## 10. robinmastromarino.com
**Что увидел:** Скролл идёт вправо (горизонтальный), во время самого скролла картинка искажается как кривое зеркало, а когда скролл останавливается — искажение пропадает, картинка становится нормальной.
**Техническое название:** Horizontal scroll + velocity-based WebGL distortion (displacement shader, привязанный к скорости скролла, гасится при остановке).
**Стек:** Three.js/OGL шейдер с uniform "scrollVelocity", при velocity→0 распад искажения через lerp.

## 11. eliastinchon.com
**Что увидел:** Сбоку — заблюренная мини-карта/индикатор, показывающий, в какой части сайта ты сейчас находишься (типа путеводителя).
**Техническое название:** Scroll progress minimap / sidebar navigation indicator.
**Стек:** фиксированный sidebar с blur-фоном, прогресс высчитывается через scroll position / общая высота, активная секция подсвечивается.

## 12. danielspatzek.com
**Что увидел:** Телевизор посреди сцены на "дороге" — переключаешь пункты меню, и картинка внутри телевизора меняется. Когда долистываешь меню — контент "залетает" внутрь экрана телевизора, и кажется, что телевизоров уже несколько на той же дороге.
**Техническое название:** 3D object as content frame + scroll-triggered scene progression (объект-контейнер для контента + сценарная анимация по скроллу).
**Стек:** Three.js/R3F, экран телевизора — отдельная плоскость с video-texture/canvas-texture, на которую рендерится UI; сцена прогрессирует по ScrollTrigger.

## Общий вывод по приёмам, которые тебе нравятся

Если сгруппировать — у тебя явная склонность к трём вещам:
1. **Нестандартное направление скролла/навигации** (горизонтальный скролл, снэп до "доминирующего" блока, 3D-сцена вместо ленты) — пункты 2, 4, 5, 7а, 9, 10, 12.
2. **Курсор и текст как живые объекты** (частицы за курсором, текст реагирует на курсор, кастомный курсор) — пункты 1, 3, 6.
3. **Переходы между состояниями как "раскрытие"**, а не просто fade (wipe, clip-path, заливка фоном, ЧБ→цвет) — пункты 1, 7б.

Это сильно WebGL/GSAP-тяжёлый набор — почти всё выше требует Three.js или как минимум GSAP ScrollTrigger. Это нормально для портфолио-сайта (он сам по себе "вечный pet-проект", не e-commerce с конверсией по миллисекундам), но важно решить архитектуру заранее, а не наслаивать эффекты один на другой без плана.

## Архитектура сайта (предложение)

**Навигация:** горизонтальный скролл через GSAP ScrollTrigger (`horizontal: true`) или Lenis + transform translateX. Вертикальный скролл мыши/трекпада конвертируется в горизонтальное движение секций.

**Структура секций (слева → справа):**
1. **Hero** — имя/бренд + один сильный тяжёлый эффект (см. ниже). Здесь происходит "вау".
2. **О себе** — короткий блок, кто ты и что делаешь (разработчик-предприниматель, свои SaaS + разработка на заказ).
3. **Кейсы** — твои проекты (Citizly, dengi и т.д.) — карточки с приёмом раскрытия (ЧБ→цвет, wipe-переход) при клике/наведении.
4. **Услуги и цены** — конкретика: что делаешь, от каких цен, как происходит процесс.
5. **Контакты/CTA** — Telegram, форма, финальный аккорд.

**Мобильная версия:** горизонтальный скролл отключается → секции идут вертикально друг под другом, тяжёлые WebGL-эффекты заменяются на статичные/лёгкие версии (через `matchMedia` и `prefers-reduced-motion`).

### Распределение эффектов по бюджету "1–2 тяжёлых + лёгкие"

**Тяжёлый эффект #1 (главный, в Hero)** — выбери один из:
- Частицы за курсором (thomasaufresne.com / mvp-code.ru) — наименее рискованный по производительности из тяжёлых.
- ЧБ→цвет раскрытие образов из частиц (benjaminrobinet.com) — эффектнее, но сложнее в реализации.

**Тяжёлый эффект #2 (опционально, в блоке кейсов или контактов):**
- Velocity-based искажение картинки во время горизонтального скролла, гаснущее в покое (robinmastromarino.com) — логично сочетается с горизонтальной навигацией, технически это один шейдер с uniform на основе скорости скролла.

**Лёгкие приёмы (не WebGL, GSAP/CSS — можно набрать сколько угодно):**
- Скролл-снэп до "доминирующего" блока (thomasmonavon.com) — CSS scroll-snap + JS-порог.
- Курсор влияет на текст рядом (clarissemichard.com) — SplitText + расчёт расстояния.
- Кастомный курсор (clarissemichard.com / общий приём).
- Мини-карта прогресса по сайту сбоку (eliastinchon.com) — просто progress-bar с blur-фоном.
- Параллакс с разной скоростью слоёв при скролле (evanfasquelle.com) — чистый CSS transform/JS, без WebGL.
- Маска с независимым движением контента внутри (kentatoshikura.com) — overflow:hidden + GSAP, без шейдеров.

Это уже даёт 2 ярких WebGL-момента + 5–6 более лёгких приёмов — сайт ощущается "богатым на эффекты", но рендер-нагрузка остаётся управляемой.

## ГЛАВНАЯ ИДЕЯ САЙТА (от Сэма) — сценарий загрузки и Hero

Это переопределяет "тяжёлый эффект #1" из раздела выше — теперь это не просто частицы за курсором, а целая сцена-сценарий.

### Этап 1 — Loading screen: планета + метеорит
- На экране вращается планета (стилизованная, минималистичная — собрана из точек/частиц, не реалистичный 3D-объект).
- Слева в кадр влетает метеорит, **на нём отображается процент загрузки сайта** (прогресс встроен в объект сцены).
- Метеорит долетает и врезается в планету.
- В момент столкновения планета **разлетается на частицы** по всему экрану.

### Этап 2 — Hero: частицы как живой фон
- Осколки/частицы от взрыва становятся постоянным фоном Hero-секции.
- Слегка двигаются сами (лёгкий дрейф/шум), не статичны.
- Постепенно появляется текст — название/оффер/описание.
- **Курсор создаёт эффект "сферы"/выпуклости** в частицах — ближе к курсору частицы приподнимаются/увеличиваются сильнее, дальше — слабее (falloff-эффект, типа smoothstep).

### Этап 3 — Переход на портфолио: частицы складываются в образы
- При горизонтальном скролле точки формируют **волну**, идущую в направлении скролла.
- Волна "перелетает" на следующий экран — точки покидают текущую секцию.
- К моменту, когда пользователь долистает, точки уже складываются в **силуэт/лого конкретного проекта портфолио**.

### Почему это сильная концепция
Все этапы используют **одну и ту же систему частиц** (`THREE.Points` с переключаемыми целевыми позициями: сфера → разброс → дрейф-облако → силуэт/лого) — весь сайт ощущается как единое целое, а не набор разных эффектов. Это снижает сложность реализации по сравнению с тремя отдельными WebGL-сценами.

### Решение по бюджету эффектов
- Отказаться от отдельного "тяжёлого эффекта #2" (искажение при скролле) — система частиц уже достаточно сильна как главный "вау"-момент.
- Лёгкие приёмы (скролл-снэп, мини-карта прогресса, параллакс) остаются как дополнение.

### Технический риск
Производительность на слабых устройствах — нужно тестировать количество частиц отдельно на мобильных, возможно урезать сцену до облегчённой версии.

## Палитра

- **Фон:** чёрный с минимальным оттенком синевы, супер тёмный (примерно `#020308`–`#05070D`, подбирается на глаз в браузере).
- **Точки/частицы:** белые/светло-серые, двухцветный набор для эффекта объёма.
- Без яркой акцентной краски — палитра минималистичная.

## Переход Hero → Портфолио: волна, а не мгновенный морфинг

При горизонтальном скролле точки не телепортируются на месте — они формируют волну, которая "перелетает" на следующий экран и успевает сложиться в силуэт/лого нужного кейса к моменту, когда экран полностью встанет на место. Реализуется через `THREE.Points` + промежуточная кривая движения (синусоида/волна по X, привязанная к прогрессу GSAP ScrollTrigger 0→1).

## Паттерн кейс-секции: логотип из точек + встроенный мини-стенд проекта

1. Волна точек складывается в **логотип конкретного проекта** (не абстрактный силуэт).
2. Параллельно формируется **«экран презентации»** — псевдо-устройство/рамка (паттерн "телевизор на дороге", danielspatzek.com), внутри которой демонстрируется контент.
3. Под экраном — **меню**, заданное Сэмом (например: "Главный экран", "Процесс оплаты", "Демо видео").
4. При выборе пункта меню содержимое внутри "экрана презентации" меняется (скриншоты/видео).

**Нужно подготовить заранее для каждого кейса:** (а) лого/контур проекта в виде, подходящем для морфинга точек, (б) набор скриншотов/видео и подписи к пунктам меню.

## Оценка сложности и производительности (мобильные устройства)

**Ключевые ориентиры:**
- Реальное количество точек — в диапазоне **5,000–20,000** (не миллионы), достаточно для убедительного визуала и управляемо по производительности.
- Морфинг (расчёт целевых позиций под лого/силуэты) считается один раз при переключении кейса, не каждый кадр.
- На мобильных: меньше точек, без курсор-эффекта (тач вместо мыши), горизонтальный скролл скорее всего заменяется на вертикальный (конфликт с системными жестами iOS Safari).

**Принятое решение по мобильным:** облегчённая версия той же системы частиц, а не отдельный другой стиль.

## Подход к разработке: Cursor vs Claude Code

Основную WebGL/частицы-логику делать через **Claude Code** (агентный режим). Cursor — для рутинной доводки после того, как ядро работает (тексты, CSS-правки, интеграция контента).

**Рекомендуемая последовательность итераций (не всё одним промптом):**
1. Статичная сфера из точек на тёмном фоне.
2. Анимация разлёта (плотный шар → широкий разброс), привязанная к loading progress.
3. Дрейф + курсор-эффект "пузыря" в покое.
4. Горизонтальный скролл + волна, уносящая точки к следующему экрану.
5. Морфинг в лого/силуэт конкретного кейса.
6. Шаблон "экран презентации" + меню под каждый кейс.
7. Мобильная облегчённая версия (после того как десктоп устраивает).

Самые нетривиальные места: синхронизация волны со скроллом и расчёт целевых позиций точек под лого/силуэты кейсов.
